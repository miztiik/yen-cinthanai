# UI Shell

**Last Updated**: 2026-07-07

Screens, components, glyphs, tokens. Mid-tier Android portrait (~390x844). Tailwind for chrome only; canvas/board internals excluded. Glyph packs are auto-discovered from the asset tree (the GlyphManifest; 12+ and growing). Visual clarity (colour-is-one-signal), not a11y tooling.

### Design rationale (Command Bar)

The board header (v2) is a single slim Command Bar line - `[home] | the DAY | timer + attempt ring + hint | tier chip + gear` - that wraps ONLY the live-solve cluster to a slim second zone under a hairline when the header is too narrow to hold one row (a container query on the header wrapper, NOT the shipped fixed two-tier), with the vertical padding cut so the bar stops eating the board. It is blur-free (no backdrop-blur) so it holds 60fps on the mid-tier Android (Holy Law #2). LEAVE is a new `ui.home` house glyph, not the old back arrow, so it can never twin the day-nav carets (the caret-grammar rule). Difficulty collapses to a SINGLE current-tier chip (bars + name from `sm+` + a down-caret) that opens the existing `DifficultyPicker` on every breakpoint - the noisy desktop 4-tier `DifficultySegmented` is deleted. Attempts read as an `AttemptRing` - `total` discrete arc wedges (one per life) whose lit arcs ramp through the config `chrome.attemptColors` urgency ramp - vivid green (full) -> amber (middle) -> red (last life) - and whose spent arcs dim to `ink` - instead of the flat `AttemptPips` dots, so the urgency of a nearly-spent budget reads at a glance; a burned arc fades its opacity over `chrome.attemptFadeMs` (transform/opacity only) while the urgency RECOLOUR of the survivors SNAPS (a discrete one-paint event, never a per-frame colour / filter / stroke-dashoffset tween - Carmack), and reduced-motion makes the fade instant. The ring vanishes on an unlimited tier. Each icon control keeps its desktop `Tooltip` (dwell = `chrome.tooltipDelayMs`, focus shows it instantly) and the calendar `DayPicker` stays off the day label. Ring arcs + tier bars are tokenized SVG / `<span>` chrome (the `TierMeter` precedent, the ResultCard progress-bar lineage), not new glyphs, and the ring knobs (`chrome.attemptFadeMs` + the `chrome.attemptColors` urgency ramp - a fixed traffic-light green/amber/red like `difficulty.colors`) live in `config/ui.json`, so Holy Laws #6/#10/#11 are untouched. The live-solve cluster (timer | ring | hint) is divided by hairline separators so the three meters read distinctly, and the whole menu is set in a thinner, smaller Jost face (distinct from the Inter board body) so the chrome recedes. Authority: User (v2 direction: slim line, home glyph, single chip, urgency ring); Jony (form); Palm (ring over hearts, discrete arcs); Carmack (blur-free frame budget, snap-recolour); Fowler (config tunable + no persisted-schema change; one cohesive PR).

### Rejected alternatives (Command Bar)

- Fixed two-tier header (the shipped Command Bar) - reads too tall/heavy and wastes vertical space the board needs; v2 is one slim line that wraps only the live-solve cluster when it must. (User; Player)
- Desktop inline 4-tier `DifficultySegmented` - the User's main offender; a single current-tier chip + the existing `DifficultyPicker` sheet carries the same choice with far less noise. (User; Jony; Player)
- Hearts / lives for attempts - needs a new baked heart glyph and carries a "lives/damage" tone against the no-shame ethos; the ring carries the count + urgency with zero new asset. (Palm; User)
- Quarter-filled / smooth-sweep ring - the User wants DISCRETE arcs (one arc per life), not a continuous donut fill. (User)
- Keeping the back arrow for LEAVE - a bold arrowhead twins the day-nav carets; a house glyph is a different shape, so the caret grammar stays legible. (User; Jony)
- CHECK left in the thumb zone (an earlier take) - reversed by the User: both RESET and CHECK/SUBMIT now ride in the Command Bar (a spent cap swaps CHECK for RETRY), and the CHECK feedback pill was re-homed DIRECTLY under the header so a first "not yet" stays adjacent to the button. The below-the-fold double-tap that burns an attempt was the reason to keep button + feedback together, so they travel together to the top rather than being split. (User)
- backdrop-blur / animated `filter` / conic-angle / `stroke-dashoffset` ring - a per-frame repaint on the mid-tier Android (Holy Law #2); the ring animates per-arc opacity only and the urgency recolour snaps.

## Route grammar

The whole URL surface is `/` (landing), `/stats`, `/settings`, and the play routes. The canonical play route is DATE-FIRST: `/play/<date>/<tier>` (`<date>` = `YYYY-MM-DD` UTC, `<tier>` in easy|standard|sharp|expert), aligned with the served file `puzzles/<date>-<tier>.json` and the save slot key `<date>|<tier>|<shapeId>` (`state/save.svelte::dayKey`). `/play/<tier>` and bare `/play` are retained ALIASES meaning "today's <tier>" / "today's next playable tier"; on load the Board resolves the day and unfurls the address bar to the dated form via `lib/router.svelte::syncLocation` (a `replaceState` that touches only the address bar - no remount, no BACK trap). Parsing (`lib/play-route.ts::parsePlay`) is order-tolerant; the builder (`playPath`) is always date-first. A dated link to a day the shipped bank no longer holds (aged past the rolling window, or a future date) falls back to the newest puzzle for that tier with a quiet notice - never a hard error. `TODAY` is UTC everywhere (loader, Stats, save), so the dated permalink is device-clock-independent and offline/share-stable. Day-to-day movement plus the "which day am I on" label live on the Board (see Screens).

### Design rationale

The URL is a contract read by the player (bookmark/share) and by the next build (resolve). Date-first makes the URL speak the bank's own key language (`date` + `tier`), the only two source-of-truth segments; `puzzleId`/`scenarioId`/`shapeId`/`file` stay derived and never appear in the URL. The dated permalink is the offline-stable form: a bookmarked dated link resolves from a stale cached bank index, whereas the bare alias dereferences through the device clock and can hard-fail at the UTC day boundary offline. Authority: User (day navigation + linkable dated URL), Carmack (canonicalize via replaceState; UTC determinism), Fowler (additive grammar; aliases grandfathered).

### Rejected alternatives

- Tier-only URL `/play/<tier>` as canonical - not linkable to a specific day and inverts the date-first layout the served files + save key already use. Kept only as an alias. (User)
- Opaque `puzzleId` in the URL (`/play/<tier>/<puzzleId>`) - a second source of truth that rots against the bank index when ids re-mint; no reverse id->file index exists. (Fowler)
- Hash routing (`#/play/...`) - uglier share URLs, murkier service-worker navigation; the existing `404.html` SPA mirror already boots history deep links. (Carmack)

## Feedback (transform/opacity)

satisfy ring+check 180ms; violate slash+shake 120ms; near amber pulse; locked solid. Easy names clue, Standard count, Sharp/Expert binary. On CHECK (any non-realtime tier) the reveal is constraint-derived, never the solution (no spoiler): a violated clue is flagged red in the ClueList and the list scrolls to the first, and any TICKED grid cell that participates in a violated clue is ringed in the reserved `--violate` colour + a corner dot (`lib/validate.ts::PuzzleEval.conflicts` = the operand-pair cells of each violated clue; painted only on ticks, only when `revealed`). The CHECK pill names it (`N clues off - shown in red`) rather than a bare "keep deducing". A hint reveals on the player's SURFACE - it ticks the anchor-vs-attribute grid cell (not a hidden token placement) and that cell pulses (scale + accent ring, `chrome.hintFlashMs`, transform/opacity) and scrolls into view, so a hint landing among filled cells is never a silent counter drop. The attempts `AttemptRing` ramps its lit arcs through the config `chrome.attemptColors` ramp - vivid green (full) -> amber (middle) -> red (last life) - as tries deplete; a just-spent arc fades its opacity over `chrome.attemptFadeMs` while the urgency recolour of the survivors SNAPS (a discrete one-paint event, never a colour tween). reduced-motion -> opacity-only / instant fade. Colour never alone (the ring also carries the count via its lit-arc tally + centre numeral + `role=img` aria-label).

## Clue glyphs

eq A=B | neq A/B | ends [A.]/[.A] | adjacent A-B | distance A>k>B | before A>>B | opposite A<>B | between A-|-B. Text under glyph teaches vocab; generator uses only types with a legible glyph.

## Story-first clues (Expand)

A story-first manifest (has `story`) swaps the glyph ClueChip strip for TEXT. StoryPanel renders the narrative cold-open; ClueList renders a numbered `<ol>` of full-sentence clues (glyphs/flags decorate the board axis headers only, never the clue language). Each ClueRow carries a trailing check-dot the player taps to strike that clue - a manual, reversible bookkeeping mark (clues start bright; never required, never nagged; LOCAL this session, persisted in Row 7 via `DayState.notes.struckClues`). The manual check-dot is the SOLE strike: a satisfied clue does NOT auto-strike (`config/ui.json [clue].autoDimFeedback` ships EMPTY), because auto-striking a clue the instant the board happens to satisfy it takes the player's own bookkeeping verb away and reads as the game playing itself (a "satisfied-right-now" clue is not a solved one - it flips back if a mark changes). The auto-dim GATE survives as config-only optionality: adding a soft feedback dial (easy realtime-names, standard count-wrong) to `autoDimFeedback` re-enables auto-dim for that dial, and sharp/expert (binary-check/submit-binary) are refused so withheld feedback could never leak. The chrome labels live in `config/copy.json [clues]`. Legacy (no-story) manifests keep the glyph ClueChip strip. Layout is chrome-only: a pull-down disclosure on phone, an always-open side panel to the right of the board on desktop. a11y: semantic `<ol><li>`, each dot a labelled `<button aria-pressed>`, visible focus ring, keyboard reachable, a polite live region announces each strike. See `src/lib/clues.ts` + `src/components/{StoryPanel,ClueList,ClueRow}.svelte`.

### Design rationale (no clue auto-strike)

The strike is the PLAYER'S "I have used this clue" verb - a small, reversible, satisfying bookkeeping action, and (per the results + scratch record below) the clue-strike IS the note-taking. Auto-applying the heavy strike visual (line-through + 50% opacity + check) on a transient `satisfy` state made the affordance mean two things at once (player bookkeeping AND live state-report) and stole the verb. So the shipped default is auto-strike OFF; the manual dot stands alone. The gate is kept (not deleted) because it is one config line and still carries the no-leak guarantee for any future opt-in. Authority: Player (I did not cross that out - confusing); Palm (the strike is the player's verb / a satisfied clue is not a solved clue); Jony (one affordance, one meaning); Fowler (config-off is the reversible step; keep the gate as cheap optionality). Rejected: deleting the gate entirely (forecloses the no-leak opt-in for no real saving); a distinct quiet "satisfied" cue that does not strike (still a feedback leak on a logic grid, and the user asked for it OFF).

## Grid axis order

The grid AXIS values render in a STABLE display order so the player scans a consistent ladder instead of the manifest's random sample order: a numeric/ordinal axis ascends by the value's `magnitude` (so "9 boxes / 3 boxes / 13 boxes / 5 boxes" reads "3 / 5 / 9 / 13"), a nominal axis ascends A->Z by `label` (natural-numeric aware), with a stable `id` tiebreak. The sort is applied once at the single choke point `src/lib/grid.ts::gridCategories` (via the pure `orderedValues` helper) on a shallow clone, so the manifest is never mutated and every grid view (GridMatrix headers + cells, NotesGrid, GridMap) reads the same order.

### Design rationale (grid axis order)

Axis order is CHROME, not puzzle content: every cell/solution lookup is id-keyed (`cellKey`/`blockCells`/`mergeGridPlacements`), so re-ordering an axis is purely visual and a correct solve still wins. The sort is done at RENDER time, not baked into the generator/manifest, because baking would churn every golden puzzle's bytes and dent reproducible builds for a display-only concern (`magnitude` already ships in the manifest). The subject/anchor axis is EXEMPT - its value order is entity-coupled (value i -> entity i), so re-sorting it would desync the identity axis; and the numbered clue list is left untouched because clue order can carry narrative/teaching meaning. Authority: User + Palm (scannable ladder); Fowler (id-keyed lookups make render-sort logic-safe); Player (leave the clue order).

## Grid crosshair (hover + keyboard focus)

Hovering OR keyboard-focusing any grid cell washes that cell's whole ROW of cells, whole COLUMN of cells, and its two axis `<th>` headers, so on a glyph axis the player instantly sees which pairing a cell asserts. It runs on BOTH grids: the phone one-block `NotesGrid` and the desktop fused `GridMatrix`. One `hovered` `$state` per grid is the single source of truth, set by FOUR delegated handlers on the scroll container (`pointerover` + `focusin` set it, `pointerleave` + a grid-leaving `focusout` clear it) - so mouse hover and keyboard focus produce the IDENTICAL crosshair (a11y parity). Handlers read `dataset` only (`data-r`/`data-c` on each `GridCell`), never `getBoundingClientRect`, so a pointer sweep triggers no forced reflow. The predicate is the pure `grid.ts::onCrosshair(hovered, r, c)`. The wash is painted in `app.css` from the `--ink` token via `color-mix` (so it follows a live theme swap): the cell wash is a `::before` overlay (`z-index -1` inside the cell's own `isolate` stacking context) that paints ABOVE the cell background but BELOW the glyph/X and leaves `box-shadow` free for the focus ring; the two headers get a direct `background-color`. Colour-only paint, no transition (a sweep must not spawn N tweens), reduced-motion-safe. The only tunable is intensity (`config/ui.json [grid.crosshair] {cell, header}`, percent of `--ink`; the header is a touch stronger since it is the primary cue), wired via `lib/crosshair.ts` (the `lib/ambient.ts` applier pattern).

### Design rationale (grid crosshair)

`hovered` is STAMPED with the block id in NotesGrid (`{block, r, c}`) and read through a `hov` `$derived` valid only for the current block, so a block switch AUTO-clears the crosshair with NO reset effect. A reset `$effect` that WROTE `hovered` was rejected: in Svelte 5 an effect that writes a signal re-runs when that signal changes, so it clobbered the live crosshair back to `null` on every hover (an `untrack` of the writes did not stop the re-run). GridMatrix needs no stamp - a new puzzle remounts it. Rejected alternatives: a pure-CSS `:has()` column highlight (unsupported below Chrome 105, a real risk on the 2022 mid-tier Android target, and cannot cleanly reach BOTH axis headers, and would duplicate `:hover` vs `:focus-within`); a `box-shadow` cell wash (would override the Tailwind focus-ring box-shadow on the focused cell, which is ALWAYS a crosshair cell - so the ring would vanish); a 9th palette token (large blast radius for a translucent ink tint - a `color-mix(var(--ink))` needs no token and trips no `app-tokens.test.ts` mirror). Authority: Carmack (no-reflow, baseline, colour-only paint); Fowler (single source of truth, config seam, no write-back effect); Jony (neutral wash, four-state legibility, headers stronger).

## Desktop grid growth + fill-the-viewport layout

On desktop (`>= lg`) the board is a fill-the-width row: the fused `GridMatrix` GROWS in the centre, with a bounded right rail (clues + live results + scratch pad) using the space that used to be an empty gutter - "this is a gaming app, not a text website". The grid grows because Board measures the grid column with a `ResizeObserver` (the layout-sized wrapper; `contentRect` only, never `getBoundingClientRect` - the same no-forced-reflow invariant as `drag.ts`) and sizes each cell via the pure `lib/fit.ts::fitCellSize(width, leafCount, cfg) = clamp(minCell, floor((width - lead - slack) / leafCount), maxCell)`. The old desktop grid floated tiny because `GridMatrix` was never passed a `size` (stuck at its 40px default); Board now threads `size={gridSize}`. Params live in `config/ui.json [grid.desktop] {minCell, maxCell, leadRem, slackPx}` + `[layout] {maxWidthPx}`; the cell is CAPPED at `maxCell` (shipped 104) so a grown cell never balloons past a comfortable box, while `maxWidthPx 0` keeps the BOARD full-viewport width (the freed width flows to the rails + centering, not into a giant cell). The axis-label font and the inter-cell gap are no longer fixed: `[grid.label] {scale, min, max}` and `[grid.gap] {scale, min, max}` TRACK the cell edge via `lib/fit.ts::scaleClamp(cell, cfg) = clamp(min, round(cell * scale), max)` (Board threads `labelPx` + `gapPx` to both grids), mirroring the GlyphSeat `d = cell * ratio` pattern - so labels and breathing space stay proportional to the box at every size. The phone one-block `NotesGrid` + `GridMap` is untouched (it reads the same `labelPx`/`gapPx` off its own cell size); the side panels become collapsible disclosures below it.

### Design rationale (desktop grid growth)

Measure the CONTAINER, not `window.innerWidth` (the clue rail shares the row, so `vw` over/under-shoots and can overflow) and not CSS container-query units (same Chrome-105 baseline the crosshair already rejected, and they still need a JS-written leaf count). The `(width - lead) / leafCount` term guarantees the fused staircase never overflows above `minCell`; below it, the existing `overflow-x-auto` scrolls. Uncapped growth (`maxCell 0`) was the user's earlier "truly fill any monitor" call, but the uncapped cell grew to ~180px while the axis labels stayed a fixed 12px and cells sat edge-to-edge - so the chrome read as broken (a ~112px glyph circle over a 12px word). Revised (2026-07-08): CAP the cell (`maxCell 104`) so labels/glyphs stay proportional, and route the "fill any monitor" intent through the BOARD width (rails + centering) rather than a ballooning cell; the axis-label font and inter-cell gap now SCALE with the cell (`scaleClamp`) so they track the box within the capped range. `maxCell` remains a one-line knob (set it to 0 to restore uncapped). The hardcoded `68/54/44` desktop cell literal, the fixed `text-xs` axis label, and the `border-spacing-0|1` gap are all retired to config (Holy Law #6). Authority: Carmack (ResizeObserver, no reflow; a 250px cell is mostly whitespace - cap it); Palm (scan-comfort cap restored); Player (words must not be ants beside giant pictures); User (revised uncapped -> capped after seeing the result); Fowler (config seam, reversible one-number knob); Jony (label + glyph are one header unit - scale together). Rejected: keep uncapped and scale labels UP to match (the vertical column label overflows the fixed `h-28` header and grows unbounded - capping the box is the cheaper, bounded fix).

## Live results roster + scratch pad

The right rail (desktop) / a disclosure stack (phone) adds two Standard+-only surfaces, both OFF the Easy cold-open (the first-60-seconds stays frozen). The LIVE RESULTS roster is the post-win `AnswerSummary` promoted to a `partial` mode: it fills each entity's attributes as the player deduces them, showing a muted `-` (+ sr-only "unknown") for the rest and a `(k/n)` count. It is fed by the pure `answer.ts::playerGrid(board, placements, ticks)` = `answerGrid` over the player's OWN merged marks (`mergeGridPlacements`), NEVER the solution - so it can never spoil, and it never forward-chains a cross-block inference the player has not made. One `$derived` in Board keys on `placements` + `gridTicks`, so Svelte memoizes it to once-per-commit and the rAF timer (which writes only `elapsed`) can never re-derive it. Desktop shows it always-open (the user's call); phone collapses it. The SCRATCH PAD is a plain `<textarea>` (`ScratchPad.svelte`) in a collapsed `<details>` that never autofocuses / raises the keyboard on load; it persists to `DayState.notes.scratch` (additive-optional, `schemaVersion` stays 1, read-default `""`), written by a DEBOUNCED setter (~400ms idle + on blur) because `localStorage.setItem` serializes the whole save and a per-keystroke write would block the main thread.

### Design rationale (results + scratch)

Both reuse the metadata-driven ideal: the results roster is ONE `AnswerSummary` with two feeds (solution on the win screen, player marks live), not a bespoke `ResultsGrid`. Sourcing the live grid from ticks (never `answerGrid(solution)` filtered) is the no-spoiler invariant. Palm argued CUT the scratch pad (the grid X and the clue-strike ARE the note-taking) and OFF the results on desktop (the fused grid already shows every tick); the user overruled both (ship the pad; results always-open on desktop), so Palm's positions survive as the collapsed/optional/off-Easy defaults + this record. The scratch debounce is idle-persistence, not game-loop timing, so `setTimeout` is permitted here. Authority: User (ship both, desktop-open); Palm (Standard+, off Easy, collapsed defaults); Fowler (one component two feeds, additive schema); Carmack (ticks-fed `$derived` memoized off the timer, debounced write).

## Glyph assets (no inline SVG)

Every icon ships as a file under `frontend/public/assets/glyphs/<pack>/<name>.svg`, referenced `"pack.slug"`. Icons come from MANY sources (varied size/weight/paint - NOT all monochrome currentColor), so the UI normalizes them visually with the Puck (below), never by assuming one paint model. The manifest `glyphs/index.json` is GENERATED every build by `tools/bake_glyphs.py` (GlyphManifest v2: auto-discovers every folder + svg; `slug` = the `pack.slug` id, `file` = the descriptive kebab filename, `label` = derived). `src/lib/glyphs.ts::glyphPath`/`glyphLabel` resolve base-aware (`/yen-cinthanai/`); `Glyph.svelte` is the ONLY image renderer - components reference ids, never inline SVG. See [../architecture/contracts/schemas.md](../architecture/contracts/schemas.md) GlyphManifest v2.

## Puck (the circular normalizer) + drag magnet

Heterogeneous-source icons are normalized by seating each inside ONE standard circle - the Puck - so the eye groups by the repeated outer shape, not the mismatched interiors. Token, filled slot, and seat are all Pucks. The logic-grid AXIS HEADERS reuse the same idea through a static, context-free sibling, `GlyphSeat` (a fixed circle + the medium Puck inset), on BOTH axes of BOTH grids (GridMatrix + NotesGrid), so a column of animals or crops aligns identically to the rows regardless of each SVG's own artboard margins - the same no-mix `glyphComplete` gate applies (an axis shows seats only when EVERY value has art, else text). Size is config-driven (`config/ui.toml [puck]`, no hardcoded px) and a player setting `puckSize` (small | medium | large, Settings > Look) scales the circle AND the inner glyph together (safe-area inset < 0.707 so the glyph never touches the rim); the Board provides it to every Puck via context. The state ring (selected/satisfy/violate/near/locked) lives in the border so it never reflows the grid. Dragging eases the puck toward the nearest valid slot centre once within capture range (`[snap]`: radius_factor x diameter + ease) and highlights that slot - a near-enough release snaps home (forgiving on a phone); capture/ease are pure + unit-tested, transform/opacity only, and tap-token-then-tap-slot stays the fallback. Two interaction invariants keep the drag honest on real devices: the glyph `<img>` is non-draggable (`draggable=false` + `-webkit-user-drag:none`, `Glyph.svelte` + `app.css`) so the browser's native image-drag can never hijack the pointer-drag on a laptop, and an OS-reclaimed pointer (`pointercancel` - scroll/zoom takeover, palm-reject) aborts the drag cleanly (no tap, no drop, capture + lifted chrome released) so a token never sticks mid-drag on touch.

## Display mode

A clarity toggle set (`settings.display{color,glyphs,labels}`, Row 2 contract) reachable from BOTH Settings > Display and an in-puzzle gear in the Board header island (the `DisplaySheet` bottom sheet, mirroring `DifficultyPicker`). `glyphs` off forces every grid axis to its text fallback (via `lib/grid.ts::axisGlyphs` gating the no-mix `glyphComplete` decision, read from a `display` context the Board provides); `color` off adds `.display-mono` to the board `<main>`, neutralizing the state/accent hue tokens to ink so the surface reads monochrome while state stays legible via ring shape + check/slash glyph + position (colour-is-one-signal). Invariant: at least one of `glyphs`/`labels` stays on, enforced in both the Settings toggles and the sheet. The same `updateSettings({display})` writer backs both entry points; there is one reader (the context) and no duplicated state.

## Components (metadata-driven)

Puck (the universal circular frame), GlyphSeat (the Puck principle at header scale - seats a value glyph in a fixed circle so a logic-grid axis of heterogeneous-source art aligns identically regardless of source SVG), Token, Slot, SlotBoard, Pool, GridMap, NotesGrid, GridCell, ClueChip, StoryPanel, ClueList, ClueRow, ActionBar, ResultCard, ShareBar, StatTile, SettingsRow (+ BottomSheet primitive). No per-screen bespoke; one Puck is the single circle used by token + filled slot + seat. Story-first manifests render StoryPanel + ClueList (text) + the cross-out grid (GridMap + NotesGrid) as the primary surface; the token SlotBoard + Pool remain only for a shared-cardinality column.

## Tailwind tokens

colors bg/surface/ink/accent/satisfy/violate/near/gold (CSS vars, palette-swappable); space 4-32, targets >=48; radii 8/16; motion 120/180 transform+opacity. Config-driven (D16).

## Ambient shell + difficulty badge

The whole-app background is a fixed, composited two-layer aurora (app.css): layer A (`body::before`) slowly transform-drifts; layer B (`html::before`) is a warmer bloom set whose OPACITY cross-fades over minutes so the palette appears to shift imperceptibly - transform + opacity only, so an animation that runs forever costs the compositor ~nothing (never `background-position` / `@property`-stop / `filter`, which repaint a full-viewport layer every frame). Both layers are token-tied (restyle on a palette swap); the timing (`driftSeconds` / `shiftSeconds` / `shiftDepth`) is config-driven in `config/ui.json [ambient]`, applied by `lib/ambient.ts` as CSS custom properties; reduced-motion zeroes it to a static warm field.

Difficulty is a colour-coded `TierMeter` (ascending bars, `rank` filled in the tier's colour) - the bar COUNT carries the level. The badge shows the tier NAME alongside the bars wherever there is room (the landing pill; the board Command Bar chip from `sm+`) plus a hover `title`; on the board the tier is a SINGLE current-tier chip (bars + name + a down-caret) that opens the `DifficultyPicker` sheet on every breakpoint (there is no desktop inline segmented switch), and the name is dropped only on the tight phone header, where the aria-label + the sheet's names carry it for a screen reader (colour never alone). It is chrome (tokenized `<span>` bars, the ResultCard progress-bar precedent), NOT a glyph icon, so Holy Law #10 is untouched. Order + per-tier colour live in `config/ui.json [difficulty]`; the colours are the cross-app STANDARDIZED difficulty ramp (green #22c55e -> yellow #eab308 -> orange #f97316 -> red #ef4444, shared with the sibling app). The `DifficultyPicker` sheet lists every tier as name + bars, opened from the single current-tier chip on both phone and desktop. The wordmark face (Josefin Sans, thin, uppercase) and its rainbow gradient are likewise standardized with the sibling app.
